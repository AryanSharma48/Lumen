# Engineering Reflection

### 1. The hardest bug you hit this week, and how you debugged it
The most challenging bug I encountered involved the Gemini API integration in the `/api/summary` route mysteriously truncating responses mid-sentence. Initially, my hypothesis was that the Vercel serverless function was timing out before the LLM could finish streaming, so I increased the `maxDuration` configuration. When that failed, I assumed my system prompt was enforcing an invalid strict character limit that confused the model. Neither was true. After wrapping the execution to log the raw API response payloads instead of just the parsed text, I noticed the `finishReason` was returning as `MAX_TOKENS`, despite me explicitly setting `maxOutputTokens: 200` for what was supposed to be a mere 3-sentence summary. 

The breakthrough came when I looked into the token usage logs. I realized that the model was consuming a significant portion of its output allowance on hidden "thinking" or "reasoning" steps before it even began generating the final text response. Because I had a strict 200-token limit to keep costs down, the model was using ~150 tokens to "think," leaving only 50 for the actual output, causing the truncation. I fixed this by raising `maxOutputTokens` to 800 to allow adequate room for the model to "show its work" while maintaining the short 3-sentence constraint through the system prompt itself.

### 2. A decision you reversed mid-week, and what made you reverse it
Mid-week, I completely reversed the core algorithmic approach of the `engine.ts` pricing evaluator. Initially, I designed the engine using a "short-circuit" pattern. It would iterate through a list of potential savings rules, and the moment it found a match (e.g., "User has Copilot and Cursor, recommend dropping Copilot"), it would apply the savings and immediately return the result. My hypothesis was that this would keep the engine incredibly fast and predictable. 

However, as I added more complex rules, like checking API efficiency versus seat-based redundancy, I realized this approach was deeply flawed for a B2B product. A user might trigger a minor $20/month savings rule first, causing the engine to short-circuit and completely miss a $500/month Enterprise consolidation rule that existed further down the execution stack. I reversed this decision and rewrote the engine to use a "candidate-collection" pattern. Instead of returning early, the engine now evaluates *every* single rule independently, collects all valid savings candidates into an array, and then reduces that array to find the optimal, highest-value recommendation. This shift from a greedy algorithm to an exhaustive search ensured mathematical accuracy and maximized the financial impact of the generated report, directly aligning with the business goal of uncovering high-value leads.

### 3. What you would build in week 2 if you had it
If I had a second week, my primary focus would shift from zero-to-one MVP development to enterprise-grade scalability and team-based workflows. Architecturally, the current synchronous LLM call in the serverless route is a liability. I would implement an asynchronous background worker using Inngest or Upstash QStash to decouple the Gemini API execution from the user request, utilizing Server-Sent Events (SSE) to stream the summary to the client. This would completely eliminate Vercel timeout risks during high-traffic spikes. Secondly, the current silent honeypot is insufficient for viral distribution. I would implement Upstash Redis rate limiting at the Edge to protect the Supabase database and Resend email quotas from coordinated bot scraping. 

On the product side, the input form currently assumes a single engineering manager knows the exact API spend for their entire organization. In reality, this data is heavily siloed. I would build a "Team Audit" flow where the primary user can invite department heads via magic links to input their specific branch's AI spend. The engine would then aggregate these discrete inputs into a master organizational report, unlocking much larger, cross-departmental efficiency insights that are currently missed by the single-player flow.

### 4. How you used AI tools
I utilized LLMs extensively as strategic sounding boards and pair programmers. I relied on AI for initial architecture brainstorming, specifically mapping out the edge cases of serverless database connections and crafting the highly constrained, zero-fluff system prompts required for the final executive summary output. I also used AI to quickly scaffold repetitive boilerplate, like Tailwind UI component shells and TypeScript interfaces for the vendor data. 

However, I maintain a strict policy of zero-trust when it comes to business logic and real-world domain data. The most glaring example of this occurred when mapping out the hardcoded pricing for the `engine.ts` logic. The AI confidently generated documentation stating that the Google Gemini Ultra tier was priced at an accessible $30 per month. Because this data is the foundational math of the product's value proposition, I manually audited the vendor sites. I discovered the AI was hallucinating based on outdated consumer pricing. The actual Enterprise-grade Ultra tier is $249.99/month, while a separate Plus tier exists at $7.99. I caught this error, provided the official pricing screenshots to correct the context window, and manually updated the core constants. AI writes the syntax; I verify the math.
